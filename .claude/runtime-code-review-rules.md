# Runtime Code Review Rules

本文件定义 Runtime 仓库代码审查的通用规则，供 `runtime-review` 和 `gitcode-pr` 技能共同使用。

## 审查维度

### A. 功能正确性

1. **逻辑正确**：变更逻辑是否符合预期意图，有无遗漏分支、边界条件未处理
2. **资源管理**：内存分配/释放是否配对，锁的获取/释放是否对称，RAII 是否正确使用
3. **并发安全**：多线程场景下共享数据是否正确保护，有无竞态条件或死锁风险
4. **API 契约**：函数前置/后置条件是否满足，参数校验是否充分（仅外部输入需校验）
5. **返回值处理**：调用链上的错误返回值是否正确传播，有无吞掉错误的情况

### B. 格式规范

1. **clang-format 一致性**：变更代码是否符合仓库 `.clang-format` 配置（120 列限制、4 空格缩进、Google 风格基底）
2. **命名规范**：
   - 函数名：大驼峰 + 尾部下划线用于私有方法（如 `DefaultStream_()`）
   - 变量名：小驼峰（如 `maxStrCount`）
   - 常量/宏：全大写下划线分隔（如 `BASE_AICPU_STREAM_ID`）
   - 类名：大驼峰（如 `DevProperties`）
3. **版权头**：新文件是否包含华为 CANN Open Software License 版权头
4. **include 顺序**：是否符合项目惯例（项目头文件 > 系统头文件）
5. **多余代码**：有无调试代码、注释掉的代码、未使用的 include/变量残留

### C. 日志合理性

1. **日志级别**：
   - `RT_LOG_ERROR`：仅用于不可恢复的错误，不可用于正常分支
   - `RT_LOG_WARNING`：功能降级、非致命异常
   - `RT_LOG_INFO`：关键操作节点（创建/销毁资源、配置变更等）
   - `RT_LOG_DEBUG`：调试信息，不应在正式代码中残留过多
2. **日志内容**：
   - 是否包含足够的上下文信息（关键变量值、ID、状态）以便定位问题
   - 是否避免打印敏感信息（内存地址仅在 debug 级别打印）
   - 格式化字符串与参数类型是否匹配（`%u` 对应 `uint32_t`，`%d` 对应 `int32_t` 等）
   - 双引号中的日志内容不能出现函数名
   - 英文单词拼写不能出现错误
3. **日志频率**：热路径中是否存在高频日志导致性能问题（循环内、高频调用函数内应避免 INFO 及以上级别日志）
4. **错误日志宏**：是否正确使用 `COND_RETURN_ERROR_MSG_INNER`、`RecordErrorLog` 等项目内错误日志宏

### D. 错误处理

1. **宏使用**：是否正确使用 `COND_RETURN_ERROR_MSG_INNER`、`COND_RETURN_WARN` 等错误处理宏
2. **错误码**：返回的 `rtError_t` 错误码是否语义准确（如 `RT_ERROR_INVALID_VALUE` vs `RT_ERROR_FEATURE_NOT_SUPPORT`）
3. **资源泄漏**：错误路径上已分配的资源是否正确释放
4. **空指针检查**：解引用前是否对可能为空的指针进行了检查

### E. 安全性

1. **缓冲区安全**：数组/缓冲区访问是否有越界风险
2. **整数溢出**：算术运算是否存在溢出风险，类型转换是否安全
3. **输入校验**：外部输入（用户 API 参数、驱动返回值）是否充分校验
4. **权限**：新增文件权限是否合理（源码 644，可执行脚本 755）
5. **未初始化变量**：是否使用了未初始化的变量
6. **数组下标检查**：数组使用前是否对数组下标做了合法性检查
7. **成员指针释放**：释放对象前，对象中的成员指针是否已释放
8. **内存初始化**：内存申请后是否需要初始化
9. **栈变量地址**：禁止返回栈空间的地址（如返回局部变量指针）
10. **锁嵌套**：普通的锁禁止嵌套（避免死锁风险）
11. **运算符优先级**：避免依赖运算符的默认优先顺序，应使用括号明确表达意图；函数入参中不能出现 `func(a++, a)` 这样的代码，计算顺序不确定
12. **内存申请释放匹配**：`new` 和 `delete`、`malloc` 和 `free`、`new[]` 和 `delete[]` 必须配对使用，禁止混用
13. **newnothrow**：使用 `new(std::nothrow)` 而非 `new`；若使用 `new` 则禁止做 `nullptr` 检查（`new` 失败会抛异常而非返回 `nullptr`）
14. **循环边界校验**：循环边界需要校验，防止死循环；边界值来自外部入参时，必须校验

### F. 公共 API 兼容性

1. **头文件变更**：`include/` 目录下的头文件修改是否保持向后兼容
2. **接口签名**：公共 API 函数签名是否发生了 breaking change
3. **废弃处理**：接口废弃是否通过标记（如 `__attribute__((deprecated))`）而非直接删除
4. **枚举兼容性**：枚举是否出现 `XX_MAX`，是否影响多版本匹配的兼容性
5. **目录规范**：`pkg_inc/runtime/runtime` 目录不应该新增内容，确认是否应该在 `pkg_inc/runtime/rt_external` 相关文件添加

### G. 文档

1. **API 文档同步**：修改了 `include/external/acl/acl_rt.h` 文件时，必须检查是否同步修改了对应的 `docs/` 目录文件；未修改则提出 **[必须修改]** 的检视意见
2. **错误码文档同步**：修改了 `include/external/acl/error_codes/rt_error_codes.h` 文件时，必须检查是否同步修改了 `docs/03_api_ref/25_数据类型及其操作接口.md` 文件；未修改则提出 **[必须修改]** 的检视意见

### H. 软件架构

1. **src/runtime 目录隔离**：非 `api` 目录不应调用 `api` 目录的方法，例如 `feature` 目录的代码不应访问 `Api::Instance`

### I. 构建规范

1. **编译宏限制**：`src/runtime` 目录仅允许使用以下编译宏：`STATIC_RT_LIB`、`RUNTIME_API`、`CFG_DEV_PLATFORM_PC` 及编译器内置宏；禁止新增其他编译宏

### J. UT 规范

1. **EXPECT 断言**：禁止缺少 `EXPECT` 断言，每个测试用例必须有断言验证结果
2. **Mock 对象清理**：每个测试类都应该 `verify` 或 `reset` mock 对象，确保测试隔离
3. **私有成员访问**：减少访问类的私有成员变量；不能包含非必要的头文件；减少 `#define private public` 和 `#define protected public` 的使用
4. **SetChipType 调用**：减少在 UT 中调用 `SetChipType`
5. **指针成员变量保护**：类的成员变量是指针时，减少重新赋值；如果需要修改，则必须记录原始内存，测试退出时恢复原有指针值

## 输出格式

按文件组织审查结果，每个问题包含：

```
### <文件路径>

- **[严重程度]** <审查维度> | 行号：<行号>
  <问题描述>
  <修改建议>（如适用）
```

严重程度分为：
- **[必须修改]**：功能缺陷、安全漏洞、资源泄漏等必须修复的问题
- **[建议修改]**：格式不规范、日志可优化等建议改进的问题
- **[仅供参考]**：风格偏好、可选优化等非必要建议

### 审查总结

在所有文件审查完成后，输出总结：

1. 变更概要：本次变更的目的和范围
2. 问题统计：各严重程度的问题数量
3. 总体评价：代码质量评估及是否建议合入
